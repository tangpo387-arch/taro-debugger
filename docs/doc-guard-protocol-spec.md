# Documentation Guard Protocol Specification (WI-52)

> [!NOTE]
> **Source Work Item**: Workflow Integration: Doc-Guard Protocol
> **Description**: Integrate doc-guard.js into Specification and Review handoff processes.

## 1. Purpose

The `doc-guard.js` tool is designed to automate the creation and verification of project documentation. This protocol defines the mandatory points of integration within the Work Item lifecycle to ensure 100% compliance with the Documentation Authoring Rules.

## 2. Integration Points

### 2.1 Specification Initialization (Product_Architect)

When the **Complexity Gate** is passed (Size M+ or architectural change), the `Product_Architect` MUST utilize the `init-spec` subcommand instead of manual file creation.

**Command**:

```bash
node scripts/doc-guard.js init-spec <WI-ID> <type> [Filename]
```

**Types**:
- `feature`: Standard feature specifications.
- `service`: Technical architecture or service-layer documentation.
- `guide`: How-to or procedural documentation.
- `troubleshoot`: Root cause analysis and resolution steps.

**Benefits**:
- Automatic frontmatter generation.
- Pre-population of Work Item context (Title, ID, Description).
- Standardized section headers to ensure complete coverage.

### 2.2 Documentation Verification (Lead_Engineer / QCR)

The `verify` subcommand MUST be executed as a mandatory pre-check before any Review Package is submitted.

**Execution Boundary**:
- **Lead_Engineer**: MUST run `lint:docs` (aliased to `node scripts/doc-guard.js verify`) before producing the Review Package.
- **Quality_Control_Reviewer**: SHOULD run `verify` at the start of a review to ensure baseline compliance.

**Criteria for Pass**:
1. Filename kebab-case (with exceptions for WI IDs).
2. Presence of mandatory YAML frontmatter (title, audience).
3. Strictly US English (no Chinese characters).
4. Correct heading hierarchy (no skipped levels).

## 3. Tool Reference

| Subcommand | Usage | Target Audience |
| :--- | :--- | :--- |
| `init-spec` | Initialize a new specification file | Product_Architect |
| `verify` | Lint documentation for compliance | Lead_Engineer, QCR |

## 4. Implementation Rules

1. **Automation-First**: Never create a `docs/` spec file manually if a template exists in `doc-guard.js`.
2. **Standardization**: Do not remove mandatory sections generated by `init-spec` unless they are explicitly marked as optional in the schema.
3. **Zero Tolerance**: Any document failing `verify` is considered "Not Deliverable" and must be fixed before QCR review.
