## 3. Edge Cases & Design Decisions

- **Dynamic GDB Re-spawning**: Intercepts client-side `initialize` request over an open kept-alive socket, kills any existing GDB subprocess dynamically first, and spawns a fresh GDB instance. This avoids closing the WebSocket connection and allows the frontend client to re-launch debug sessions seamlessly.
- **WebSocket Keep-Alive**: The daemon remains active even when GDB exits to permit multiple sequential debug sessions. The daemon cleanly exits with exit code 0 *only* when the client WebSocket closes.
- **Orphan Process Sweeper**: If the client disconnects, `src/server.ts` cascades cleanup: it calls `await this.gdbProcess.terminate()`, stops all sockets, closes the file streams, and terminates the daemon process via `process.exit(0)`.
- **Fail-Fast Port Checking**: The port checks will fail-fast and throw clean exceptions if the designated port is already occupied.
- **DAP Content-Length Parser**: The stream parser in `src/gdb-process.ts` accumulates chunks and splits them precisely using GDB-native `Content-Length: <size>\r\n\r\n` header delimiters to prevent JSON parsing of half-packets.
- **`disconnect()` Visibility (Addendum)**: `disconnect()` is now `private` in `DapSessionService`. External callers — including `ngOnDestroy()` in `DebuggerComponent` — must not call it directly. The correct public API for session teardown is `stop()` (active sessions) and the internal terminated-event chain handles the final disconnect step automatically.
- **`disconnect()` Error-State Semantics (Addendum)**: `disconnect()` now treats `error` state as a safe early-return (same as `disconnected`), because the error state implies the transport is already closed and there is nothing to disconnect cleanly.
- **🔍 Inspect — `cancelRequest()` Guard (Addendum)**: The `disconnected` state guard was added to `cancelRequest()` (line L886–890) to prevent sending cancel requests over a closed transport. QCR should verify that the three-condition guard (`!== 'idle'`, `!== 'disconnected'`, `!== 'error'`) is consistent with the service's transport lifecycle.
