## 3. Edge Cases & Design Decisions

- **Optimistic Continuation**: Implemented an optimistic state toggle to `running` within `continue()`. If the adapter confirms `allThreadsContinued: true` in the response body, we instantly update the `executionState` and clear `stoppedThreads`. We also reset `commandInFlight` immediately in this case to ensure the "Pause" button re-enables without waiting for the `continued` event.
- **Command Blocking (R-CS1)**: I extended the `commandInFlight` window for stepping commands. Instead of resetting it in `finally`, it now stays `true` upon a successful DAP response and is only reset when the subsequent `stopped` or `continued` event is handled. This prevents the `notStopped` error seen in the user's report by blocking the UI until the transition is truly complete.
- **Error Handling**: If a control command fails (e.g. invalid state), `commandInFlight` is reset immediately to allow the user to recover.
- **Thread ID Fallback**: Used `activeThreadId || 1` for control commands. This ensures that even if no thread is explicitly selected in the UI (or before the first `threads` fetch), we still attempt to step the primary thread.
- **Visual Density**: The "ACTIVE" badge is placed on the right side of the thread label (`margin-left: auto`) to avoid cluttering the thread name/ID and to maintain a high-density, professional look.
- 🔍 **Inspect**: The `commandInFlight` handling in `DapSessionService.ts`. I verified via `repro-issue.spec.ts` that it correctly handles the race condition where a `stopped` event arrives before the command response.
